# ADR-011: Task CRUD HTTP API

## Status

Accepted — 2026-07-09

Implements `docs/M1-VERTICAL-SLICES.md` Slice 2 (Task CRUD API).

## Context

The CLI already provides task creation, inspection, spec freeze, and state transitions. M1 adds an HTTP API so the web board and future clients can drive the same lifecycle without shelling out to the CLI.

The API must preserve Marshal's current source-of-truth model:

- SQLite owns mutable task state.
- The task state machine owns valid transitions.
- Spec freeze at Ready creates the frozen spec commit and worktree.
- Slug generation must stay consistent with the CLI.

The main architectural risk is accidentally creating a second task-control path with behavior that diverges from existing CLI commands.

## Decisions

### 1. Expose task lifecycle operations as JSON HTTP endpoints

The daemon will expose:

| Method | Path | Purpose |
| --- | --- | --- |
| `GET` | `/api/tasks` | List tasks with board-card fields. |
| `GET` | `/api/tasks/:slug` | Return one task with full detail. |
| `POST` | `/api/tasks` | Create a backlog task. |
| `POST` | `/api/tasks/:slug/transition` | Move a task through a valid state-machine edge. |
| `POST` | `/api/tasks/:slug/ready` | Freeze or update the spec and transition the task to Ready. |

Responses are JSON only. The first M1 API does not include pagination because task volume is expected to be small and local to one repo daemon.

### 2. Reuse store and lifecycle functions directly; never shell out to the CLI

HTTP handlers call the same task store, state machine, slug-generation, and spec-freeze helpers used by CLI commands. They must not execute `marshal task ...` as a subprocess.

If existing CLI logic is trapped inside command handlers, Slice 2 should extract reusable functions into the task module rather than duplicate behavior in HTTP handlers. The CLI and HTTP layer then become two thin adapters over the same domain functions.

### 3. Keep response shapes stable and board-oriented

`GET /api/tasks` returns the fields needed for the board:

```json
{
  "tasks": [
    {
      "id": 1,
      "slug": "example-task",
      "title": "Example task",
      "status": "backlog",
      "retry_count": 0,
      "created_at": "2026-07-09T00:00:00.000Z",
      "updated_at": "2026-07-09T00:00:00.000Z"
    }
  ]
}
```

`GET /api/tasks/:slug` returns the same fields plus full detail:

```json
{
  "task": {
    "id": 1,
    "slug": "example-task",
    "title": "Example task",
    "status": "backlog",
    "spec_markdown": "...",
    "retry_count": 0,
    "last_failure": null,
    "created_at": "2026-07-09T00:00:00.000Z",
    "updated_at": "2026-07-09T00:00:00.000Z"
  }
}
```

Mutation endpoints return the updated or created task in the same detail envelope.

### 4. Use strict request validation and consistent error envelopes

Request bodies reject unknown fields and invalid values. Errors use one envelope:

```json
{
  "error": "Invalid transition from backlog to done",
  "code": "invalid_transition"
}
```

HTTP status codes:

| Status | Use |
| --- | --- |
| `400` | Malformed JSON or unknown fields. |
| `404` | Task slug not found. |
| `409` | Valid JSON, but lifecycle operation conflicts with current task state. |
| `422` | Semantically invalid body, such as an unknown status. |
| `500` | Unexpected daemon error. |

Unexpected errors should be logged by the daemon and returned without stack traces.

### 5. `POST /api/tasks` creates backlog tasks only

Task creation accepts:

```json
{
  "title": "Implement X",
  "spec_markdown": "Optional draft spec"
}
```

The created task starts in `backlog`. The slug is generated by the same normalization and collision-handling logic as the CLI. Empty titles are rejected.

### 6. `POST /api/tasks/:slug/ready` owns spec freeze semantics

The Ready endpoint mirrors `marshal task ready`. It accepts:

```json
{
  "specMarkdown": "Optional replacement spec"
}
```

If `specMarkdown` is provided, it replaces the stored draft before freeze. If omitted, the stored `spec_markdown` is frozen. The endpoint returns the updated task after the freeze and transition complete.

The freeze operation remains atomic from the client's perspective: if writing the spec, committing the frozen artifact, or creating the worktree fails, the API returns an error and must not pretend the task is Ready.

### 7. HTTP transitions use the same state machine as CLI transitions

`POST /api/tasks/:slug/transition` accepts:

```json
{
  "to": "review"
}
```

It delegates to the existing `transitionTask` behavior, including escape-hatch retry reset from ADR-009. The HTTP API does not introduce new state edges.

## Consequences

- The web board can drive the task lifecycle without calling CLI subprocesses.
- CLI and HTTP behavior stay aligned because both use the same task domain functions.
- Strict JSON validation gives web clients predictable failures and avoids accepting typo-shaped state changes.
- The first API is intentionally repo-local and unpaginated; scale concerns are deferred until there is real task volume.
- The Ready endpoint becomes the first HTTP route with filesystem and git side effects, so its error handling must be explicit and tested.

## Open questions (deferred)

- **Field casing.** The first API mirrors existing database field names (`spec_markdown`, `retry_count`) in responses while accepting `specMarkdown` for the Ready override because the slice names that body field. A later public API pass may standardize everything to camelCase.
- **Pagination and sorting.** Task list pagination is deferred until local board performance requires it.
- **Bulk operations.** M1 keeps one task at a time; no bulk transition or creation endpoint is added.
