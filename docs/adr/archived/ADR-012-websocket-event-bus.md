# ADR-012: WebSocket Event Bus

## Status

Accepted — 2026-07-09

Implements `docs/M1-VERTICAL-SLICES.md` Slice 3 (WebSocket Event Bus).

## Context

The M1 web board needs live task and run updates without polling. Slice 2 gives clients a REST API for lifecycle operations, but a board that polls for every state change would either feel stale or create unnecessary load on the local daemon.

`docs/PROJECT.md` §3.1 names WebSocket as part of the daemon API contract. Slice 3 defines the first real-time contract: connect to `/ws`, receive an initial task snapshot, and receive task/run/daemon events as they happen.

M1 is localhost-only and single-user in practice, but multiple browser tabs and CLI/web clients may be connected at once.

## Decisions

### 1. Use a small internal event bus as the daemon's real-time boundary

The daemon will expose an internal publish/subscribe interface for domain events. Task store operations, run logging, and orchestrator cycle events publish to this bus after durable state changes complete.

WebSocket delivery is one subscriber to that bus. This keeps real-time behavior out of domain code and leaves room for future subscribers, such as logs, tests, or a TUI client adapter.

### 2. Use WebSocket at `/ws` on the same daemon port

The WebSocket endpoint is:

```text
ws://127.0.0.1:7433/ws
```

It attaches to the same underlying Node server selected in ADR-010. HTTP routing remains Hono-owned; WebSocket upgrade handling is attached separately to the server's `upgrade` path.

Implementation should use the `ws` package unless a later implementation spike finds a smaller maintained library with equivalent Node upgrade support.

### 3. Send a connected event with an initial task snapshot

On successful connection, the daemon immediately sends:

```json
{
  "type": "connected",
  "payload": {
    "tasks": []
  },
  "timestamp": "2026-07-09T00:00:00.000Z"
}
```

The task objects in the snapshot match `GET /api/tasks` from ADR-011. This lets a client initialize from one stream without racing a separate HTTP list request against live events.

Clients may still call `GET /api/tasks` for explicit refresh. The connected snapshot is a convenience and synchronization point, not a replacement for the REST API.

### 4. Standardize event envelopes

All WebSocket messages use:

```json
{
  "type": "task.created",
  "payload": {},
  "timestamp": "2026-07-09T00:00:00.000Z"
}
```

`type` is a dot-separated string. `payload` is an object. `timestamp` is an ISO-8601 string generated by the daemon when the event is published.

The initial M1 event types are:

| Event type | Payload |
| --- | --- |
| `connected` | `{ tasks }` |
| `task.created` | `{ task }` |
| `task.updated` | `{ task }` |
| `task.transitioned` | `{ task, from, to }` |
| `run.started` | `{ run }` |
| `run.event` | `{ runId, event }` |
| `run.finished` | `{ run }` |
| `daemon.idle` | `{}` |
| `daemon.cycle_complete` | `{}` |

Task payloads use the same task shape as ADR-011. Run payloads use the run-log shape from ADR-006 and later Slice 4's run-history API.

### 5. Broadcast only after durable writes succeed

Task events are emitted after SQLite commits. Run events are emitted after the run log writes. This ensures a client that receives a WebSocket event can immediately fetch the referenced task or run through HTTP and see consistent state.

If an operation fails before persistence, no success-shaped WebSocket event is sent. Errors from HTTP mutations are returned to the initiating client through the HTTP response, not broadcast as global events.

### 6. Support multiple clients with heartbeat-based cleanup

The daemon supports multiple concurrent WebSocket clients. Each connection receives every broadcast event after it connects.

Heartbeat policy:

- Server sends ping every 30 seconds.
- Client must respond with pong.
- Server drops a client after 90 seconds without a pong.

Dropped clients do not affect the daemon loop or other clients.

### 7. No replay buffer in Slice 3

Slice 3 does not provide event replay, sequence numbers, or resume tokens. A client that disconnects reconnects to `/ws`, receives a fresh `connected` snapshot, and continues from there.

Run-event pagination arrives through Slice 4's HTTP API, not through WebSocket replay.

## Consequences

- The board can update task cards and run output in real time without polling.
- The daemon gains a clear internal event boundary that domain code can publish to without knowing about sockets.
- Clients get a simple recovery model: reconnect and accept the latest snapshot.
- Because events are broadcast after durable writes, WebSocket and HTTP stay consistent.
- Event replay, delivery acknowledgements, and cross-process fanout are deferred as unnecessary for one local daemon process.

## Open questions (deferred)

- **Event versioning.** M1 event types are stringly typed. If external clients begin depending on them, introduce a versioned schema.
- **Backpressure.** Slice 3 assumes event volume is low. If run output overwhelms browsers, add throttling or per-client send queues in Slice 4.
- **Authentication.** No WebSocket auth is added while the daemon is localhost-only. Non-localhost exposure requires the same future auth decision as HTTP.
